# Evaluation Criteria

| Criteria | Description |
|---|---|
| Problem Definition | 부산시 교통 문제를 데이터 기반 예측 문제로 정의했는가 |
| Data Quality | 데이터 출처, 컬럼, 결측치, 이상치 처리 기준이 명확한가 |
| WBS | 작업 단위, 담당자, 일정, 산출물이 구체적으로 분해되었는가 |
| Database Design | 교통량, 기상, 대기질, 공휴일, 예측 결과를 저장할 DB 구조가 적절한가 |
| ERD | 주요 엔티티와 관계를 명확하게 표현했는가 |
| SQL Schema | PK, FK, 인덱스, 데이터 타입이 적절하게 설계되었는가 |
| Modeling | 예측 모델을 학습하고 성능을 비교했는가 |
| Visualization | 사용자가 이해할 수 있도록 결과를 시각화했는가 |
| Web Service | Streamlit으로 예측 결과를 서비스 형태로 구현했는가 |
| Collaboration | GitHub Issue, Commit, 역할분담 기록이 남아 있는가 |
| Reflection | KPT 회고를 통해 문제점과 개선 방향을 도출했는가 |

## 5-Level Rubric

| 평가기준 | 매우미흡 | 미흡 | 보통 | 우수 | 매우우수 |
|---|---|---|---|---|---|
| 문제정의 | 문제를 단순 코딩 과제로만 이해함 | 부산 교통 문제를 일부 설명함 | 부산 교통량 예측 목표를 설명함 | 지역·날씨·공휴일 요인을 연결함 | 부산 도시 특성과 예측 필요성을 근거로 설명함 |
| 데이터 | 출처와 컬럼이 불명확함 | 일부 데이터만 수집함 | 기본 데이터셋을 구성함 | 결측치·이상치 처리 기준을 기록함 | 데이터 품질 문제와 처리 근거를 명확히 제시함 |
| WBS | 작업 분해가 없음 | 큰 단위만 나열함 | 작업·담당·산출물을 구분함 | 일정과 Sprint를 연결함 | GitHub Issue까지 연결하여 관리함 |
| DB/ERD | DB 설계가 없음 | 테이블만 단순 나열함 | 주요 테이블과 관계를 제시함 | PK/FK와 데이터 사전을 제시함 | 학습 이력·예측 결과까지 포함한 확장 구조를 제시함 |
| 모델링 | 모델 실행 결과가 없음 | 모델 실행은 했으나 평가 부족 | 1개 이상 모델 성능 제시 | 여러 모델 비교 및 선정 이유 제시 | 성능 한계와 개선 방향까지 설명함 |
| 서비스 | 실행 불가 | 일부 화면만 구현 | 기본 Streamlit 화면 구현 | 입력-예측-결과 흐름 구현 | 해석 문구와 시각화까지 완성도 있게 구현 |
| 협업·성찰 | 기록 없음 | 일부 개인 중심 기록 | 역할분담과 회고 작성 | 이슈·커밋·회고 기록 | 피드백 반영 이력까지 포트폴리오화 |
